# 四、需求不明确时必须向用户提问

当用户需求不明确、硬件型号不明确、现象描述不足、接口边界不清、目标板子不明确、预期行为不明确时，必须先向用户提问，而不是自行规划需求边界或幻想代码实现。

该规则应作为需求不明确阻断 hook 自动触发。只要缺少继续实现所需的关键条件，AI 必须先暂停实现流程，向用户索要信息；不能一边假设一边写代码。

必须提问的情况包括但不限于：

1. 用户只说"加一个驱动"，但没有说明硬件型号；
2. 用户只说"修一下 Bug"，但没有说明现象、日志、复现步骤；
3. 用户只说"接入某个模块"，但没有说明通信方式；
4. 用户没有说明目标板子或固件路径；
5. 用户没有说明功能应该放在 BSP、middleware 还是业务层；
6. 用户没有说明是否要兼容多板子；
7. 用户没有说明是否需要低功耗、NVS 持久化、唤醒、音频、网络等联动行为。

典型触发示例：

```text
用户：帮我加一个新传感器支持。
AI 必须先问：请提供传感器型号、通信接口、参考例程或数据手册。
```

硬件类需求至少需要问清：

1. 准确型号；
2. 通信接口；
3. GPIO 接线、总线编号、设备地址或中断脚；
4. 目标板子和固件路径；
5. 参考例程、数据手册或原理图；
6. 期望功能边界；
7. 是否需要兼容多板子、低功耗、唤醒或持久化配置。

Bug 类需求至少需要问清：

1. 具体现象；
2. 完整日志；
3. 复现步骤；
4. 期望行为；
5. 实际行为；
6. 目标芯片、目标板子和 ESP-IDF 版本；
7. 最近做过的修改。

禁止行为：

1. 不得在需求不明确时擅自创造硬件型号；
2. 不得擅自假设 GPIO 引脚；
3. 不得擅自假设 I2C/SPI/UART 地址或总线编号；
4. 不得擅自决定业务行为；
5. 不得生成无法验证的幻觉代码。
6. 不得在没有型号、接口、例程或数据手册时直接新增传感器驱动；
7. 不得在没有日志和复现路径时武断判断 Bug 根因。
