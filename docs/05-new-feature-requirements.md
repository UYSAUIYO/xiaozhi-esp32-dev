# 五、新功能实现需求处理规则

当用户提出新功能实现需求时，应先判断功能类型，再决定代码放置位置和实现方式。

## 5.1 新硬件驱动或新硬件支持

该规则应作为新硬件资料确认 hook 自动触发。只要用户提出新硬件驱动编写或加入新硬件支持，必须先确认是否存在官方例程、Arduino 例程、ESP-IDF 例程、寄存器手册、通信协议说明，再开始迁移代码。

如果用户提出的需求属于以下类型：

1. 新增传感器驱动；
2. 新增音频芯片支持；
3. 新增功放、Codec、麦克风、触摸芯片、IO 扩展芯片；
4. 新增屏幕、灯带、电机、电源管理芯片；
5. 新增 I2C/SPI/UART/GPIO/PWM/ADC/I2S 外设；
6. 新增某个板载硬件能力；
7. 用户说“添加 XX 芯片支持”“编写 XX 驱动”“接入 XX 模块”“支持新的传感器”；

则必须主动向用户索要以下信息：

1. 新硬件的准确型号；
2. 模块原理图或接线方式；
3. 数据手册；
4. 官方例程或厂商例程；
5. 当前项目中目标板子的名称和路径；
6. 使用的 ESP-IDF 版本；
7. 期望实现的功能边界；
8. 是否需要支持低功耗；
9. 是否需要支持多板子复用。

资料索要顺序：

1. 优先索要官方例程、厂商例程、Arduino 例程、ESP-IDF 例程或当前项目可参考的已有驱动；
2. 如果没有例程代码，必须索要数据手册、寄存器手册、通信协议说明或原理图；
3. 如果例程和资料都没有，必须停止驱动实现，只能说明缺失信息和可准备的最小资料清单。

如果用户提供了例程代码，应优先按照参考例程进行代码迁移，而不是重新发明驱动。

迁移时应分析：

1. 例程初始化流程；
2. 通信接口；
3. 寄存器配置；
4. 时序要求；
5. GPIO 依赖；
6. 中断或轮询方式；
7. 错误处理；
8. 与 ESP-IDF 组件机制的适配方式。

如果用户没有例程代码，应主动索要数据手册、寄存器手册或通信协议说明，并基于资料明确：

1. 设备通信方式；
2. 寄存器表；
3. 初始化序列；
4. 功能能力边界；
5. 不支持的能力；
6. 最佳驱动实现方式。

禁止在没有例程、没有数据手册或寄存器手册、没有硬件型号的情况下直接编写完整驱动。

## 5.2 Bug 修复需求

如果用户提出的是 Bug 修复需求，应按以下流程执行：

第一步，检查 Git 状态：

```bash
git status --short
```

第二步，阅读相关代码，但不能只读报错点或单个文件。必须理解：

1. 功能入口；
2. 调用链；
3. 初始化顺序；
4. 组件依赖；
5. 配置项；
6. 相关任务、回调、中断、事件循环；
7. 日志输出位置；
8. 是否存在类似模块或历史实现。

第三步，向用户索要调试日志信息。

应优先索要：

```text
1. 完整编译日志；
2. 完整运行日志；
3. 串口 monitor 输出；
4. 复现步骤；
5. 期望现象；
6. 实际现象；
7. 使用的目标芯片；
8. 使用的 ESP-IDF 版本；
9. 当前分支；
10. 最近做过的修改。
```

如果用户已经给出日志，应快速定位问题并提出修复方案。

如果用户没有日志，应向用户问清楚 Bug 现象，并在代码中增加必要的调试日志。

日志应使用 ESP-IDF 标准日志系统：

```c
#include "esp_log.h"

static const char *TAG = "module_name";

ESP_LOGI(TAG, "init success");
ESP_LOGW(TAG, "unexpected state: %d", state);
ESP_LOGE(TAG, "operation failed: %s", esp_err_to_name(err));
```

对于多轮调试，应优先通过日志和复现路径逐步缩小问题范围，不应一次性大规模重构。
