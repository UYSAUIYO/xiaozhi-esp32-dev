# 四、需求不明确时必须向用户提问

当用户需求不明确、硬件型号不明确、现象描述不足、接口边界不清、目标板子不明确、预期行为不明确时，必须先向用户提问，而不是自行规划需求边界或幻想代码实现。

必须提问的情况包括但不限于：

1. 用户只说"加一个驱动"，但没有说明硬件型号；
2. 用户只说"修一下 Bug"，但没有说明现象、日志、复现步骤；
3. 用户只说"接入某个模块"，但没有说明通信方式；
4. 用户没有说明目标板子或固件路径；
5. 用户没有说明功能应该放在 BSP、middleware 还是业务层；
6. 用户没有说明是否要兼容多板子；
7. 用户没有说明是否需要低功耗、NVS 持久化、唤醒、音频、网络等联动行为。

禁止行为：

1. 不得在需求不明确时擅自创造硬件型号；
2. 不得擅自假设 GPIO 引脚；
3. 不得擅自假设 I2C/SPI/UART 地址或总线编号；
4. 不得擅自决定业务行为；
5. 不得生成无法验证的幻觉代码。
